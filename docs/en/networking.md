# Networking Model

This document describes the `crius` networking model and the boundary between
local `crs` workflows and CRI/kubelet workflows.

## Network Domains

`crius` uses CNI as the default network implementation and separates local
`crs` networking from CRI networking:

| Domain | Default config directories | Used by |
| --- | --- | --- |
| `[network.local]` | `/etc/crius/cni/net.d` | `crs pod run`, `crs run --pod`, `crs container create <pod>` |
| `[network.cri]` | `/etc/cni/net.d`, `/etc/kubernetes/cni/net.d` | kubelet, `crictl runp`, CRI `RunPodSandbox` |

Legacy top-level `[network]` fields are still parsed and map to the CRI network
domain. New local `crs` Pod workflows should not inherit Kubernetes CNI
directories by default.

## Default Local CRS Pod Network

The recommended default configuration for local `crs pod` usage is:

```toml
[network.local]
config_dirs = ["/etc/crius/cni/net.d"]
plugin_dirs = ["/opt/cni/bin", "/usr/lib/cni", "/usr/libexec/cni"]
cache_dir = "/var/lib/cni/crius-local-cache"
```

Install the sample CNI configuration:

```bash
sudo install -Dm644 examples/cni/crius-bridge.conflist \
  /etc/crius/cni/net.d/10-crius-bridge.conflist
```

The configuration defines a local bridge network named `crius-bridge` and uses
the following standard CNI plugins:

| Plugin | Purpose |
| --- | --- |
| `loopback` | Enables the loopback interface in the Pod network namespace |
| `bridge` | Creates and connects the host bridge; the default bridge name is `crius0` |
| `host-local` | Allocates local IP addresses for Pods on this host |
| `portmap` | Implements hostPort mappings |

The default subnet is `10.88.0.0/16`, with gateway `10.88.0.1`. The example
enables `isGateway`, `ipMasq`, and `hairpinMode`, which covers common local Pod
egress and port publishing scenarios.

## Difference From CRI Networking

The CRI network domain is for the Kubernetes model:

```toml
[network.cri]
config_dirs = ["/etc/cni/net.d", "/etc/kubernetes/cni/net.d"]
plugin_dirs = ["/opt/cni/bin", "/usr/lib/cni", "/usr/libexec/cni"]
cache_dir = "/var/lib/cni/cache"
```

This domain serves kubelet and `crictl`. For example:

```bash
crictl --runtime-endpoint unix:///run/crius/crius.sock runp pod.json
```

The local ordinary-container path used by `crs run` does not require
`[network.cri]` to be ready. `crs pod` uses `[network.local]`. Therefore,
`crs debug network` may validly report:

```text
local network: ready
cri network: not-ready
```

This means local `crs pod` networking is available while CRI/kubelet CNI
configuration is not yet prepared.

## Calico, Flannel, And Other CNI Plugins

The default target for `crs pod` is CRS local explicit Pod networking. The
default configuration loads only the CNI directories configured by
`[network.local]`; it does not automatically inherit Kubernetes or CRI CNI
directories, and it does not assume that Calico, Flannel, kube-apiserver, etcd,
or other Kubernetes node components exist.

Calico and Flannel can be attempted through custom CNI configuration, but users
must satisfy their plugin dependencies:

- Calico commonly depends on calico-node, IPAM, the Kubernetes API, or another
  datastore.
- Flannel commonly depends on subnet information generated by flanneld and is
  usually paired with cluster node networking.
- Plugin lifecycle, routing, and address management are outside the scope of
  the default local bridge network.

The recommended default remains the standard `containernetworking-plugins`
combination:

```text
loopback + bridge + host-local + portmap
```

## Common Scenarios

Local Pod networking:

```bash
pod=$(crs --quiet pod run --name web --publish 8080:80)
crs run --detach --pod "$pod" nginx:latest
crs pod inspect "$pod"
```

Host network Pod:

```bash
crs pod run --name hostnet --host-network
```

Network diagnostics:

```bash
crs debug network
crs --output json debug network
crs doctor
```

## Runtime Requirements

Local CNI workflows usually require:

- `crius` daemon running as root.
- executable CNI plugin binaries in `plugin_dirs`.
- valid `.conf`, `.conflist`, or `.json` files in `config_dirs`.
- a daemon-writable `cache_dir`.
- host support for network namespaces, veth, bridge, iptables, or nftables.

Explicit `crs pod` networking requires a valid local CNI configuration.

## Troubleshooting

Collect these details first:

```bash
crs --output json debug network
crs --output json doctor
systemctl status crius
journalctl -u crius --no-pager
```

Common issues:

| Symptom | Possible cause |
| --- | --- |
| `local network is not configured` | `/etc/crius/cni/net.d` does not contain valid local CNI config |
| `CNIConfigMissing` | the selected network domain has no valid CNI config |
| `missing plugin binaries` | CNI config references plugins that are not installed or executable |
| Pod cannot reach external networks | bridge, IP masquerade, host forwarding, or firewall rules are incorrect |
| hostPort does not work | `portmap` is missing or the CNI config does not declare the `portMappings` capability |

## Stability

This document describes current networking behavior. `crius` remains an
experimental runtime implementation, and public network configuration fields and
diagnostic output may evolve with the project.
